ML Expressions and Variable Bindings

  • An ML program is a sequence of bindings
  • Each binding gets type-checked and then evaluated
  • What type a binding has depends on a static environment
  • How a binding is evaluated depends on a dynamic environment

There are several kinds of bindings, but for now let’s consider only a variable binding

  • Syntax: val is a keyword, x can be any variable, and e can be any expression
  • Type-checking: x having type t where t is the type of e
  • Evaluation: x having the value v where v is the result of evaluating e

Here are several different kinds of expressions:

  • Integer constants
  • Addition
  • Variables
  • Conditionals
  • Boolean constants
  • Less-than comparison

Using use

When using the REPL, it is very convenient to add a sequence of bindings from a file.

  • Its type is unit
  • Its result is ()
  • Its effect is to include all the bindings in the file "foo.sml"

Variables are Immutable

  • Bindings are immutable
  • The later binding shadows the earlier one

Function Bindings

  • A function is something that is called with arguments and has a body that produces a result
  • There is no notion of a class, this, etc
  • Do not have things like return statements

Syntax

  • This is a binding for a function named x0
  • It takes n arguments x1, ..., xn of types t1, ..., tn
  • Has an expression e for its body

Type-checking

  • A function type is “argument types” -> “result type”
  • The argument types are separated by *
  • ML can use type inference to figure out the type of e

Evaluation

  • A function is a value
  • We simply add x0 to the environment as a function that can be called later

Function calls

  • A new kind of expression
  • The syntax is e0 (e1,…,en)
  • The typing rules:
    • e0 has a type that looks like t1*...*tn->t
    • For 1 ≤ i ≤ n, ei has type ti
    • The whole call has type t
  • The evaluation rules:
    • use the environment at the point of the call to evaluate e0 to v0, e1 to v1, …, en to vn
    • Then v0 must be a function
    • We evaluate the function’s body in an environment extended
    • The environment is the one when the function was deﬁned

Pairs and Other Tuples

Pairs

  • The syntax is (e1,e2)
  • The typing rules:
    • the type of a pair is t1*t2
    • t1 is the type of the first part
    • t2 is the type of the second part
  • The evaluation rules:
    • e1 to v1 and e2 to v2
    • make the pair of values (v1,v2)
  • use #1 and #2 to access the first and second part
    • e must have some type that looks like ta * tb
    • #1 e has type ta and #2 e has type tb

Examples:

Tuples

  • ML supports tuples by allowing any number of parts
  • An example is (7,9,11)
  • You can retrieve the parts with #1 e, #2 e, and #3 e
  • Pairs and tuples can be nested however you want

Lists

  • Lists can have any length
  • all the elements of any particular list must have the same type
  • The empty list:
    • syntax []
    • It is a value
    • It can have type t list for any type t
    • ML writes as ’a list
  • A non-empty list:
    • syntax [v1,v2,...,vn]
    • More common to make a list with e1 :: e2
      • e1 evaluates to an “item of type t”
      • e2 evaluates to a “list of t values”
  • we will use three functions provided by ML:
    • null evaluates to true for empty lists and false for nonempty lists
    • hd returns the first element of a list, raising an exception if the list is empty
    • tl returns the tail of a list, raising an exception if the list is empty

Examples:

  • Functions that make and use lists are almost always recursive
  • To write a recursive function:
    • thinking about the base case
    • then the recursive case
  • we can combine pairs and lists however we want

Let Expressions

  • An absolutely crucial feature
  • It lets us have local bindings of any sort
  • It can appear anywhere an expression can

Syntactically, A let-expression is:

  • The syntax: each bi is a binding and e is an expression
  • the scope of a binding in a let-expression is:
    • the later bindings in that let-expression
    • the “body” of the let-expression
  • The type of e is the type for the entire let-expression
  • The value e evaluates to is the value for the entire let-expression

Examples:

  • Deﬁne a local function that uses other variables in scope
    • a hugely common and convenient thing to do in functional programming
  • We can use let-expressions to avoid repeated computations

Options

  • An option value has either 0 or 1 thing:
    • NONE is an option value “carrying nothing”
    • SOME e evaluates e to a value v
    • The type of NONE is ’a option
    • The type of SOME e is t option if e has type t
    • We have isSome which evaluates to false if its argument is NONE
    • We have valOf to get the value carried by SOME

Some Other Expressions and Operators

Here is a brief list of some additional forms of expressions:

  • e1 andalso e2 is logical-and
  • e1 orelse e2 is logical-or
  • not e is logical-negation
  • e1 = e2 judges if two numbers are same
  • e1 <> e2 judges if two numbers are different
  • other arithmetic comparisons: >, <, >=, <=
    • can be used with real
    • but not 1 int and 1 real
  • use ~e for negation

Lack of Mutation and Benefits Thereof

  • There is no way to change the contents of a binding, a tuple, or a list
  • Having immutable data is one of the main contributions of functional programming
  • It makes sharing and aliasing irrelevant

The Pieces of a Programming Language

Here are the essential “pieces” necessary for defining and learning any programming language

  • Syntax: How do you write the various parts of the language?
  • Semantics: What do the various language features mean?
  • Idioms: What are the common approaches to using language features to express computations?
  • Libraries: What has already been written for you?
  • Tools: What is available for manipulating programs in the language?
